# Bun Client (Phase -1 of Electrobun Migration)

Builds a v2 implementation of `IClient` that runs under Bun with no Electron
dependencies. Lives alongside the existing `src/preload/client/` — Electron
keeps working throughout. When the Electrobun phases land, the main process
switches to this client and the old one is deleted.

**Parent plan:** [docs/plans/pending/electrobun-migration.md](../pending/electrobun-migration.md)

---

## Why parallel, not in-place

The swap touches the DB driver, query layer, settings store, and image
handling simultaneously. Doing it in-place means the Electron app is broken
during the work and neither side can be tested cleanly. `IClient` is already
the clean seam — two implementations can satisfy it at once.

---

## Target layout

```
src/bun-client/
  schema.ts           # Drizzle table definitions — source of truth for DB shape
  migrations/         # SQL files generated by drizzle-kit
  factory.ts          # createClient() → IClient, uses bun:sqlite + Drizzle
  settings-store.ts   # JSON r/w replacement for electron-store
  journals.ts
  documents.ts
  tags.ts
  preferences.ts
  indexer.ts
  importer.ts
  bulk-operations.ts
  smoke.ts            # CLI entry: opens real DB, lists journals (Phase 8)
  importer.ts         # Markdown import pipeline — do last, see Phase 9
```

Shared (unchanged):

- `src/preload/client/types.ts` — `IClient`, all request/response types
- `src/preload/client/util.ts` — `createId`, `checkId`

---

## IClient interface note

`IClient` currently exposes `knex: Knex` as a public field. Tests and a few
call sites use it for direct DB inspection. The v2 client should expose
`db: BunSQLiteDatabase` (Drizzle's typed db) in the same spirit — a typed
escape hatch for tests. Update `IClient` to replace `knex: Knex` with
`db: unknown` (or a shared supertype) before starting, or simply add `db` to
the v2 client type and widen `IClient` to accept both. Decide at Phase 1.

---

## Stack

| Concern     | Choice                                 | Notes                                                        |
| ----------- | -------------------------------------- | ------------------------------------------------------------ |
| DB runtime  | `bun:sqlite`                           | Built into Bun, no native rebuild                            |
| Query layer | Drizzle ORM (`drizzle-orm/bun-sqlite`) | Only tool with native first-party bun:sqlite support         |
| Migrations  | `drizzle-kit`                          | Generates SQL files, applies programmatically                |
| Settings    | Custom JSON r/w (`settings-store.ts`)  | Replaces `electron-store`; tiny API surface                  |
| Images      | No-op / passthrough                    | Drop `sharp`; existing fallback always writes original bytes |
| Test runner | `bun test`                             | Built-in, fast, no config needed                             |

## Phases

### Phase 1 — Runtime + schema + migrations ✅

**Status:** Complete.

**Goal:** Prove Drizzle + bun:sqlite works end-to-end. No application logic yet.

**What was done:**

- Added `drizzle-orm` and `drizzle-kit` to `package.json`
- Created `src/bun-client/schema.ts` — Drizzle table definitions for all 12 tables
- Generated migrations via `bunx drizzle-kit generate`
- Created `src/bun-client/factory.ts` — opens `bun:sqlite`, applies migrations,
  creates FTS5 virtual table, returns typed Drizzle db
- Created `src/bun-client/factory.test.ts` — 5 tests, all green

**IClient decision:** Deferred widening `IClient` for now. The bun-client returns
a `BunClient` type with `db` (Drizzle) and `sqlite` (raw Database) fields. The
`IClient` interface will be unified when modules are wired up in Phase 8.

**Note — shared `package.json`:** Drizzle deps live in the same `package.json`
as the Electron app. So far no conflicts (drizzle-orm is pure JS, drizzle-kit is
dev-only). If a future dep conflicts with Electron's native rebuilds or bundler
assumptions, consider extracting `src/bun-client/` into its own `package.json`
with a workspace setup. For now, single `package.json` is simpler.

**Validation:** `bun test src/bun-client/factory.test.ts` — green (5/5 pass).

## What the test harness looks like

Each test file creates an in-memory DB and a temp `notesDir` in `beforeAll`,
tears them down in `afterAll`. No Electron, no webview, no IPC.

```ts
import { beforeAll, afterAll, test, expect } from "bun:test";
import { mkdtempSync, rmSync } from "fs";
import { tmpdir } from "os";
import { createClient } from "./factory";

let client: Awaited<ReturnType<typeof createClient>>;
let notesDir: string;

beforeAll(async () => {
  notesDir = mkdtempSync(tmpdir() + "/chronicles-test-");
  client = await createClient({ dbPath: ":memory:", notesDir });
});

afterAll(() => {
  rmSync(notesDir, { recursive: true, force: true });
});
```

## Decisions

1. **Drizzle, not Knex.** Knex has no bun:sqlite support (tracking issue open
   March 2024, no PR as of March 2026). Drizzle has native first-party
   `drizzle-orm/bun-sqlite` support and is the dominant pattern in the Bun
   ecosystem.

2. **No sharp.** Drop it entirely. The existing fallback (write original bytes)
   stays. Image resizing and webp conversion are deferred to a future pass.

3. **No electron-store.** Replace with a purpose-built `settings-store.ts`
   — a thin JSON file wrapper with dotted-path get/set/delete. Zero new deps.

4. **Parallel, not in-place.** `src/preload/client/` is untouched until
   Electrobun phases land and the switch is flipped.
